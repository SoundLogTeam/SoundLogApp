# 비개발자용 Codex 개발 가이드

이 문서는 개발자가 아니어도 Soundlog 앱을 Codex와 함께 안전하게 개선할 수 있도록 만든 작업 가이드입니다.

## 기본 원칙

Codex에게 한 번에 “다 만들어줘”라고 하기보다 아래 순서로 맡기는 것이 좋습니다.

1. 현재 상태 확인
2. 구현 계획 수립
3. 엣지케이스 질문
4. 계획 리뷰 및 보완
5. 구현
6. 타입체크와 웹 확인
7. 코드 리뷰와 수정 반영
8. 커밋 및 푸시

이 흐름을 지키면 앱이 깨졌을 때 원인을 찾기 쉽고, 디자인 의도와 기능 의도가 어긋나는 일을 줄일 수 있습니다.

## 기능 요청 방법

좋은 요청은 “어떤 화면에서, 어떤 사용자가, 어떤 결과를 얻어야 하는지”가 들어갑니다.

```text
홈 화면의 나의 무드 추천 섹션을 개선하고 싶어.
여행 모드와 무드 키워드가 겹치지 않게 하고,
필터 선택 시 추천 카드 순서가 바뀌도록 구현해줘.
먼저 계획을 세우고 엣지케이스가 있으면 질문해줘.
```

## 디자인 수정 요청 방법

디자인은 Figma 링크나 스크린샷을 함께 주는 것이 가장 좋습니다.

```text
이 Figma 화면을 기준으로 리캡 공유 페이지를 더 예쁘게 만들어줘.
카드, 배경, 공유 CTA, 저장 CTA를 중심으로 계획을 세우고 구현해줘.
웹에서 확인 가능한 상태까지 검증해줘.
```

자연어 피드백만 있는 경우에도 아래처럼 루프를 지정하면 됩니다.

```text
이 UI 수정은 비개발자 피드백 루프로 진행해줘.
내 요청을 먼저 구체적인 UI 목표로 번역하고,
계획 -> 계획 리뷰 -> 반영 -> 구현 -> 코드리뷰 -> 코드리뷰 반영 순서로 진행해줘.
```

## 서버가 아직 없을 때

Soundlog는 `src/mock-server`로 서버 기능을 흉내낼 수 있습니다. 서버 개발 전에도 홈, 플레이리스트, Recap, 주변 관광지 기능을 테스트할 수 있습니다.

로딩 상태 확인:

```bash
EXPO_PUBLIC_MOCK_API_DELAY_MS=1500 npm run web
```

실패 상태 확인:

```bash
EXPO_PUBLIC_MOCK_API_FAIL_ENDPOINTS=recap.share npm run web
```

## 검증을 요청할 때

최소한 아래 검증을 요청하세요.

```text
npm run typecheck를 실행하고,
가능하면 웹에서 해당 화면을 열어 깨지는 부분이 없는지 확인해줘.
문제가 있으면 수정하고 다시 검증해줘.
```

## 커밋을 요청할 때

구현이 끝난 뒤에는 아래처럼 요청하면 됩니다.

```text
변경사항 요약해주고, 불필요한 파일이 없는지 확인한 뒤 작업 브랜치에 커밋하고 PR 만들어줘.
커밋 메시지는 한국어로 작성해줘.
```

Soundlog는 `main`에 직접 push하지 않습니다. 모든 변경은 작업 브랜치에서 PR로 올립니다. 자세한 흐름은 `docs/codex/PR_ONLY_WORKFLOW.md`를 확인하세요.

## 주의해야 할 요청

아래 내용은 사용자 경험이나 개인정보에 영향이 크므로 Codex가 질문하고 진행해야 합니다.

- 위치 추적을 백그라운드에서도 할지
- 사진 없이 순간 로그를 저장할지
- 음악 플랫폼 연동 실패 시 앱 안에서 어떻게 보여줄지
- Recap 공유가 OS 공유 시트만 사용할지, 앱별 공유까지 지원할지
- 로그아웃 상태에서도 데이터를 로컬에 남길지
- 실제 서버 API 실패 시 캐시/Mock/Fallback 중 무엇을 우선할지
